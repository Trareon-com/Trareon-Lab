# APFS validation dossier (M8)

## Status

**CORPUS_VALIDATED** for the golden-image subset below.

## Validated subset

- `nx_superblock_t` in block zero, including the `NXSB` magic at byte 32,
  block size/count, UUID, transaction ID, object-map OID, and volume OID array.
- Fletcher-64 checksum validation for every parsed APFS object.
- Container and volume object-map lookup using a single root/leaf B-tree node.
- `apfs_superblock_t` volume index, UUID, name, and root-tree OID.
- A single root/leaf file-system catalog B-tree containing `j_inode_val_t` and
  case-sensitive `j_drec_key_t` / `j_drec_val_t` records.
- Directory/file path reconstruction, type, and inode uncompressed size.
- Truncated input and bad magic return `LabError`; they do not panic.

Evidence: `crates/lab-fs/tests/apfs_real.rs`, generated by
`write_minimal_apfs_image`, and `cargo test -p lab-fs`.

## Explicit limitations

This is not full macOS APFS feature parity. Multi-node B-trees, hashed
case-insensitive directory keys, snapshots, encryption, sealed volumes,
fusion drives, hard links, extended fields, and file-content extent reads are
not validated. Inputs requiring those paths fail closed instead of being
silently interpreted as the golden subset.

`TRFSAPFS` remains a separate Experimental synthetic format under
`lab_fs::apfs_synth`; it is not accepted as a real APFS container.
